# Chat to QA Skill

从零散的群聊对话中智能提取问答对，理解上下文语义，生成结构化的 QA 并自动分类标签。

## When to Use This Skill

- 从群聊记录中提取有价值的问答信息
- 需要理解上下文才能判断是否是 QA 的场景
- 将聊天记录转化为知识库内容
- 对 QA 进行自动分类（系统bug、数据问题、业务问题、用户问题等）
- 构建智能客服知识库

## Prerequisites

- 大模型 API 访问权限（支持上下文理解）
- 群聊消息数据（格式化的消息列表）
- 分类标签定义文件

## Instructions

### 1. 输入数据准备

输入应为格式化的聊天消息列表，包含以下字段：

```json
[
  {
    "msg_id": "消息唯一标识",
    "sender_id": "发送者ID",
    "sender_name": "发送者名称",
    "content": "消息内容",
    "timestamp": "时间戳",
    "chat_id": "群聊ID"
  }
]
```

### 2. QA 提取规则

请按照以下规则从聊天记录中提取 QA：

#### 2.1 识别问题
- 包含疑问词：怎么、如何、为什么、能不能、可以、是否、什么
- 包含问号：？、?
- 表达困惑或求助的陈述句
- 报告问题或异常的描述

#### 2.2 识别答案
- 直接回答问题的内容
- 提供解决方案的描述
- 解释原因或步骤的说明
- 确认问题已解决的反馈

#### 2.3 上下文理解
- 一条消息可能是问题，答案可能分散在后续多条消息中
- 同一话题的讨论可能跨越多条消息，需要合并理解
- 注意区分闲聊内容与有价值的问答
- 识别追问和补充说明，合并到同一 QA 对

#### 2.4 QA 质量标准
- 问题应清晰、完整、独立可理解
- 答案应准确、有用、具有参考价值
- 忽略纯闲聊、表情包、无关内容
- 对于模糊的问答，标注置信度

### 3. 分类标签体系

使用以下分类标签对每个 QA 进行标注：

| 标签 | 英文标识 | 说明 | 识别关键词 |
|------|----------|------|------------|
| 系统Bug | system_bug | 系统功能异常、报错、崩溃 | 报错、错误、崩溃、异常、bug、500、404、无法、失败、卡住、闪退 |
| 数据问题 | data_issue | 数据错误、缺失、不一致 | 数据、不一致、错误、缺失、丢失、重复、对不上、不匹配、同步 |
| 业务问题 | business_issue | 业务流程、规则、配置问题 | 流程、审批、规则、配置、设置、权限、业务、操作、步骤、办理 |
| 用户问题 | user_issue | 用户使用咨询、操作指导 | 怎么、如何、能不能、可以、怎样、使用、操作、功能、在哪里 |
| 功能需求 | feature_request | 新功能建议、改进意见 | 建议、希望、想要、新增、功能、改进、优化、需求、期待 |
| 其他 | other | 不属于以上分类的问题 | - |

### 4. 输出格式

输出的 QA 应包含以下字段：

```json
{
  "qa_list": [
    {
      "id": "qa_唯一标识",
      "question": "整理后的问题内容（清晰、完整）",
      "answer": "整理后的答案内容（准确、有用）",
      "category": "分类标签英文标识",
      "category_name": "分类标签中文名称",
      "confidence": 0.95,
      "context": "上下文说明（可选）",
      "source_msg_ids": ["关联的消息ID列表"],
      "keywords": ["关键词列表"],
      "severity": "严重级别：critical/high/medium/low"
    }
  ],
  "statistics": {
    "total_count": 10,
    "by_category": {
      "system_bug": 2,
      "user_issue": 5,
      "business_issue": 3
    }
  }
}
```

### 5. 严重级别判断

根据问题内容判断严重级别：

| 级别 | 判断标准 |
|------|----------|
| critical | 系统崩溃、核心功能不可用、影响所有用户 |
| high | 功能异常、数据丢失风险、影响部分用户 |
| medium | 一般问题、有临时解决方案 |
| low | 咨询类问题、功能建议 |

### 6. 处理流程

```
输入聊天消息
    │
    ▼
消息预处理（去噪、合并连续消息）
    │
    ▼
上下文理解（识别对话主题、参与者意图）
    │
    ▼
QA 候选识别（问题检测、答案匹配）
    │
    ▼
QA 质量评估（完整性、准确性、价值度）
    │
    ▼
自动分类标注（匹配分类规则、关键词）
    │
    ▼
严重级别判断
    │
    ▼
输出结构化 QA 数据
```

## Examples

### Example 1: 提取系统 Bug QA

**输入聊天记录：**
```
[10:00] 用户A: 页面打开就报错了，显示500错误
[10:01] 用户A: 刷新了好几次都不行
[10:02] 技术支持: 请提供一下具体的页面地址和操作步骤
[10:03] 用户A: 就是点开订单列表的时候
[10:05] 技术支持: 好的，这个问题我们查到了，是服务器临时故障，已经修复了，请再试一下
[10:10] 用户A: 好了，谢谢
```

**输出 QA：**
```json
{
  "id": "qa_001",
  "question": "打开订单列表页面时出现500错误怎么办？",
  "answer": "500错误是服务器临时故障导致的。如果遇到此问题，可以尝试刷新页面或等待一段时间后重试。如果问题持续，请联系技术支持提供具体的页面地址和操作步骤进行排查。",
  "category": "system_bug",
  "category_name": "系统Bug",
  "confidence": 0.95,
  "context": "用户访问订单列表功能时遇到服务器错误",
  "keywords": ["500错误", "订单列表", "服务器故障"],
  "severity": "high"
}
```

## Error Handling

### 常见问题处理

1. **消息内容缺失**
   - 跳过空消息或纯表情消息
   - 合并同一用户连续发送的多条消息

2. **问答不完整**
   - 只有问题没有答案：标记为待补充
   - 只有答案没有问题：尝试从上下文推断问题

3. **分类不确定**
   - 置信度低于 0.6 时标记为 "other"
   - 提供分类理由便于人工审核

4. **上下文过长**
   - 按 20 条消息为一批处理
   - 保留相邻批次的上下文衔接

## Output for Knowledge Base

生成的 QA 数据可直接导入知识库系统，支持以下格式：

- JSON 格式（推荐）
- CSV 格式
- Markdown 格式

每条 QA 记录包含完整的分类标签和元信息，便于知识库的检索和管理。
